## [新聞] GSC中索引涵蓋範圍報表出錯

最近Lily Ray在推特上提到在GSC中的「索引涵蓋範圍報表」出現錯誤，沒想到一呼百應，很多SEO(包括我自己)都也經歷過一樣的問題。

在索引涵蓋報表中顯示為「*已檢索 – 目前尚未建立索引 (Crawled – currently not indexed)*」的網址，如果使用「網址檢查工具」，卻會顯示該網頁已索引，而在Google上也能確實找到這些網頁。

對此，官方的解釋是「索引涵蓋範圍報表」本來就會以較慢的速度更新，所以數據上有差異是正常的，應該要以「網址檢查工具」為主。但有[SEO回報](https://twitter.com/jroakes/status/1448206881999183872)其網站98%的網頁都顯示未被收錄 (但其實有被收錄)，所以不像是延遲這麼簡單。


> A few Google Search Console users reported that they saw URLs in the Index Coverage report marked as "Crawled – currently not indexed" that, when inspected with the URL Inspection tool, were listed as "Submitted and indexed" or some other status. (1/4\)
> 
> — Google Search Central (@googlesearchc) [October 11, 2021](https://twitter.com/googlesearchc/status/1447531448278437895?ref_src=twsrc%5Etfw)

## [新聞] Google Search On 2021摘要

Google Search On今年來到第二屆，雖然不像[去年的活動](https://www.darrenhuang.com/seo-newsletter-issue-18.html#toc-1)在SEO界因為[passage retrieval](https://www.darrenhuang.com/passage-retrieval-and-seo.html)引起了一陣旋風，但還是能看到Google在搜尋這塊的未來展望和方向。其中的又與[今年Google I/O的MUM](http://43.201.249.211/seo-newsletter-issue-32-33.html#toc-8)(機器學習領域模型)息息相關。 

**1\. 幫助消費者找到搜尋靈感** ([文章](https://blog.google/products/shopping/search-on-2021-shopping/))  
主要提了三點，(1\.) 用Google Lens協助購物流程，使用者可以在看到一張圖片時，選定一個區域(產品)並進行搜尋；(2\.) 在Google搜尋上購物，使用[Google’s Shopping Graph](http://43.201.249.211/seo-newsletter-issue-32-33.html#:~:text=3.%20%E5%BB%BA%E7%AB%8B%20Shopping,%E5%93%81%E7%89%8C%E3%80%81%E8%A9%95%E5%83%B9%E2%80%A6%E7%AD%89%E7%AD%89)%E3%80%82)的技術，將產品、價錢、評價、存貨、品牌、風格…等等整合在一起，提供一個完整豐富的購物環境；(3\.) 整合實體購物(in\-store)，使用者可以看到所在區域附近商家的存貨情形，整合線上與現下購物體驗。

**2\. AI與搜尋的結合** ([文章](https://blog.google/products/search/how-ai-making-information-more-useful/))  
這段講的是MUM (機器學習的新里程碑)，實施在搜尋上可以帶來甚麼樣的改變。使用者將可以進行非常複雜甚至跨領域(文字vs圖片)的搜尋指令，例如對著一個花襯衫的圖片，輸入一段文字「*找到有這個圖案的襪子*」，來獲得結果。此外隨著MUM的幫助，Google也會在SERP上建議更廣義或更具體的搜尋主題，來幫助不確定自己在找什麼的使用者進行搜尋。

**3\. 搜尋上的額外資訊** ([文章](https://blog.google/products/maps/search-on-maps-2021/))  
這裡比較像PR的橋段，在談搜尋結果如何讓使用者信任，其中提到[About this result](http://43.201.249.211/seo-newsletter-issue-37.html#toc-2)提供更多關於該網站的資訊；fact check的事實查核機制避免假消息傳播；以及在[找不到相關結果](https://www.darrenhuang.com/seo-newsletter-issue-7.html#toc-2)的時候也會誠實講。

**4\. Google Maps上的進步** ([文章](https://blog.google/products/maps/search-on-maps-2021/))  
講了Google Maps上可以整合其他的資訊，例如野火區域、或者綠地覆蓋區域，已成為更多功能的工具。另外一個很有趣的是[Plus Code](https://maps.google.com/pluscodes/)這個項目，能夠幫助國家建立自己的地址系統。住在先進國家的人可能比較難想像，但很多開發中/未開發國家連一套完整的地址系統都沒有，這個[4分多鐘的影片](https://www.youtube.com/watch?v=8hFgoCVeyuw)推薦大家看看。

## [官方回應] Google如何應對網站出現像Facebook那樣的伺服器崩壞情形

十用初的[臉書大當機](https://technews.tw/2021/10/05/facebook-says-outage-was-faulty-configuration-change/)相信大家都有所耳聞，但如果當你所管理的網站遇到這樣子的情形，SEO該如何應對? Google又是如何看待網站伺服器爆炸的情形? 在Twitter上John Mueller稍微解釋了Google的觀點。

* 當網站出現伺服器沒有回應或者5XX的情形，Google首先會減少抓取的頻率
* 當這樣的情形持續了1\~2天，Google就會開始逐漸將這些網頁移除索引
* 通常這情形**不直接影響排名**，Google並不會因此覺得網頁品質低或者內容不好
* 伺服器問題修復後，網頁就會再度出現在搜尋結果中，抓取頻率也會逐漸回復


> If you're curious about what happens in Google Search with an outage like Facebook recently had, it's generally a 2\-part reaction: when we can't reach a site for network / DNS reasons, we see it like a 5xx HTTP server error. This means we reduce crawling: <https://t.co/V7CjrVq3yJ>
> 
> — 🧀 John 🧀 (@JohnMu) [October 5, 2021](https://twitter.com/JohnMu/status/1445271604288700419?ref_src=twsrc%5Etfw)

連結: [官方文檔 – 如何妥善的將網站在Google搜尋上暫停](https://developers.google.com/search/blog/2020/03/how-to-pause-your-business-online-in)

## [技巧] Google Discover的搜尋參數

我個人很愛使用搜尋參數這個小技巧，簡單說就是在Google SERP的”網址”上加入一些參數，來做到篩選的效果，舉例來說，在網址後方加入`[num=100](https://www.google.com/search?q=%E6%90%9C%E5%B0%8B%E5%BC%95%E6%93%8E&num=100)`可以讓搜尋結果一頁變成100個；加入`[lr=lang_zh-TW](https://www.google.com/search?q=%E6%90%9C%E5%B0%8B%E5%BC%95%E6%93%8E&num=100&lr=lang_zh-TW)`可以只跑出繁體中文結果…等等。

最近學到了一個新知，那就是Google Discover也可以透過參數來模擬不同情境下出現的的網頁結果。首先要將瀏覽器改成手機模式，然後修改主要三個參數 – `q`(改變搜尋詞)、`lr`(改變語言、以及`tbs=qdr:`(改變時間範圍)。

```
https://www.google.com/search?q=%E8%8B%B1%E9%9B%84%E8%81%AF%E7%9B%9F&ibp=tpf;cl_tp:srp,ui_tp:gst&lr=lang_zh-TW&tbs=qdr:w
```

## [觀念] 網址中的大小寫會影響SEO嗎

在[搜尋引擎爬蟲抓取](https://www.darrenhuang.com/how-search-engines-crawl.html#all-about-urls)這篇文章中我有寫到過類似的觀念，下面六個網址，那些在搜尋引擎的眼中是一樣的，哪些又是不一樣的?

```
 A) https://www.darrenhuang.com/about-darren-huang/ (結尾多了斜線)
 B) http://www.darrenhuang.com/about-darren-huang (開頭為http而非https) 
 C) https://www.darrenhuang.com/ABOUT-DARREN-HUANG (大寫)  
 D) https://darrenhuang.com/about-darren-huang (少了www) 
 E) https://www.darrenhuang.com/about-darren-huang?utm_source=facebook (後面多了追蹤參數) 
 F) https://www.darrenhuang.com/about-darren-huang.html (後面多了.html的檔名) 
```
在該文中有解答，而下面這個影片裡，John更詳細的解釋Google如何解讀，答案是「都不一樣」。網址使用大寫或小寫，嚴格定義上是指不同的網頁，但Google也知道這樣的情形，通常是伺服器設定上的不完美而非刻意人為所致，因此Google會自己嘗試判斷究竟是不小心還是故意的。

SEO的工作就是要減少讓Google來”猜”你的網站，不管在內容或是架構上，都是越清楚越好，影片中也有提到如何給Google清楚的訊號，避免模糊空間。

最後考大家一個問題，robots.txt中disallow的網址路徑，大小寫是否有差?

## [Podcast] Google Search Podcast – Gary與Martin爭執JavaScript

Gary Illyes和Martin Splitt這兩個人一直以來都很喜歡在JavaScript是不是個好東西上面鬥嘴，這次的podcast沒有John Mueller，兩個人就選了這個主題直球對決，討論JavaScript的起源與演化，它的好處及壞處。

這次的podcast就不幫整理摘要了，因為這個主題牽扯到一個語言的歷史演化，需要豐富的CS背景才有辦法簡潔的整理，我對JS只有小研究，不敢勝任這個腳色，但這集podcast大致講了JavaScript這個語言最最開始是個伺服器端的語言(當時叫Mocha)，後來才被拿去作為瀏覽器語言(當時稱作LifeScript)，因此它的本性其實並不是為了現今網站設計而生的語言，而是慢慢演化而成，因為一些歷史的因素(podcast中有更詳細的故事)，人們發現JS很適合被用來做一些與網站有關的東西，因此開發出像AJAX, jQuery這些延伸應用，近幾年又有向React, Angular, Vue這些用JS為底的網站建構framwork。

Martin和Gary兩位都是開發者背景出生，Gary認為JS太”低端”，定義不夠嚴謹，也容易有出乎意料的結過；而Martin不否認這點，但也同時反駁道演化就是這樣，JS有它的方便性，儘管開發者常常非常有創意地將其使用在錯誤的地方。這次的podcast我會推薦給有一些程式基礎(如果是JS就更好)或者是開發者聽，兩個人一邊拌嘴一邊討論還滿有趣的，對此完全沒有經驗可能會略顯生澀。
